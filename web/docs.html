Lovelang Documentation

A complete reference for the Lovelang interpreted language — syntax, CLI, VS Code extension, and examples. Written for humans who feel things.

v1.0.0 stable C11 runtime VS Code Extension Native Compile .love files
Introduction

Lovelang is a C-based interpreted language built around desi human-style phrases. It's a passion project born from the idea that programming languages don't have to be cold and mechanical — they can feel warm, expressive, and personal.

The language uses a Hinglish-inspired vocabulary: yaad for mutable variables, vada for constants, dhadkan for functions, and bolo to print output. It now includes modules (import/export), list/map collections, richer string helpers, conversion utilities, and filesystem built-ins.

The execution pipeline is: file read → preprocess → lex → parse → execute. No JVM, no package manager, no runtime dependencies. Just clone, make, and run.

✨ Philosophy: Mix natural style and core syntax freely. Lovelang is designed so that both yaad count hai 0 and yaad count = 0 work. Write what feels right.
Installation

Recommended: install the npm CLI wrapper. It auto-downloads the correct binary for your OS/CPU and runs immediately.

npm (macOS / Linux / Windows)
terminal
# Install globally (auto-downloads correct binary for your OS)
$ npm install -g lovelang-cli
 
# Interpret a .love file
$ lovelang hello.love
 
# Run in shayari mood
$ lovelang hello.love --mode shayari
 
# Compile to native binary (no external toolchain needed)
$ lovelang hello.love --native --out ./hello
Windows
PowerShell
# Using the PowerShell runner
PS> .\run_love.ps1 examples\hello.love
 
# Or using the cmd runner
> run_love.cmd examples\hello.love
Building from Source

The Makefile handles compilation of all source files. Source is in src/, headers in include/.

Makefile targets
$ make # build ./lovelang binary
$ make clean # clean build artifacts

Project layout:

  • include/love.h — token/AST declarations shared by all modules
  • src/main.c — CLI entry point and human-phrase preprocessor
  • src/lexer.c — tokenizer and keyword recognition
  • src/parser.c — AST builder
  • src/runtime.c — tree-walk execution engine
  • examples/ — runnable .love programs
  • extension/ — VS Code extension
Your First Program

Create a file called hello.love and write:

hello.love
// A classic greeting, Lovelang style
yaad name hai "jaan"
vada promise hai "forever"
 
baby bolo naa "hello, " milan name
 
agar name barabar hai "jaan" toh
  bolo "welcome back 💌"
warna
  bolo "who are you?"
bas itna hi
 
love you baby byeee
 
» hello, jaan
» welcome back 💌
» ❤️ byeee...

Run it:

terminal
$ ./lovelang hello.love
Syntax Overview

Lovelang supports two syntactic styles that can be freely mixed:

  • Core syntax — structured, uses curly braces and semicolons optionally
  • Natural/human style — uses desi phrases, no braces needed, ends with bas itna hi

The preprocessor runs first and normalizes human-mode phrases into core tokens before the lexer sees them.

Variables & Constants
Keyword Type Meaning
yaad mutable Declare a mutable variable. Can be reassigned.
yaad_karo mutable Alternate mutable declaration (same behavior as yaad).
vada immutable Declare a constant. Reassigning throws a runtime error.
variables.love
// Mutable declarations
yaad count hai 0
yaad_karo score hai 10
 
// Immutable constant
vada badge hai "gold"
 
// Human-mode alias for vada
baby yaad rakho name hai "jaan"
 
// Assignment (name must exist)
count hai count milan 1
 
// This would cause a runtime error:
// badge hai "silver" → immutable!
Value Types

Lovelang has six runtime value types:

Type Literals Notes
int 0, 42, -7 Integer arithmetic
bool sach, jhooth true / false in desi
string "hello" Double-quoted strings
list via list_nayi() Ordered dynamic collection
map via map_naya() Key/value dictionary (string keys)
null null Absence of value
Note: sach = true, jhooth = false. These are the only boolean literals in Lovelang.
Modules: import / export

Modules are handled in preprocessing. import inlines another file, and export marks reusable declarations in module files.

  • import "path/file.love" resolves relative to the current file
  • same import is included once per run (deduped)
  • export is accepted for readability and stripped before parsing
modules.love
import "modules/romantic_utils.love"
bolo greet("patel krish")
bolo greet("patel krish", mood = "shayari", punct = "...")

Conditionals

Two syntactic forms, same semantics:

Natural Style (Recommended)
conditionals.love
agar score bada hai 90 toh
  bolo "A grade 🎉"
warna agar score bada hai 60 toh
  bolo "passing"
warna
  bolo "try again :("
bas itna hi
Core Style
agar (score > 90) ye_karo {
  bolo "A grade";
} vo_karo {
  bolo "not yet";
}
One-Liner
// sun na agar cond toh then_stmt warna else_stmt
sun na agar ok toh bolo "yes" warna bolo "no"
Note: bewafa can be used in place of a secondary condition check (similar to else-if). It follows the same bewafa cond toh pattern.
Loops
Keyword Meaning
intezaar … tab tak while loop ("waiting until")
jabtak … tab tak while loop (alternate form)
loops.love
yaad i hai 0
 
// intezaar = "waiting" (while i < 3)

  bolo i
  i hai i milan 1
bas itna hi
 
» 0
» 1
» 2
Loop guard: A runtime safeguard triggers after 1,000,000 iterations to prevent infinite loops.
Functions

Functions are defined with dhadkan (heartbeat), return with ehsaas (feeling), and now support default params + named arguments.

functions.love
// Define a function
dhadkan greet(name, mood = "romantic", punct = "!") {
  ehsaas mood milan ": " milan name milan punct
}
 
// Positional and named calls
bolo greet("jaan")
bolo greet("jaan", mood = "shayari")
bolo greet("jaan", mood = "toxic", punct = "...")
 
// Computation function
dhadkan add(a, b = 0) {
  ehsaas a milan b
}
bolo add(5) // 5
bolo add(5, b = 3) // 8
Note: ehsaas is only valid inside a dhadkan block. Unknown named args or extra positional args raise runtime errors.
Festival Blocks

A festival block is a named block that prints a runtime header and executes its body. Great for marking celebrations in your code.

festival.love
festival diwali {
  bolo "✨ lights on"
  bolo "diyas everywhere"
}
 
» festival mode: diwali
» ✨ lights on
» diyas everywhere

Stop Behavior

Any of these forms stop program execution and print a mode-specific farewell:

Form Notes
love_byeee() Core function form
love_you_baby_byeee() Expressive alias
love you baby byeee Natural phrase (preprocessor normalizes)
love you baby bye... Trailing ellipsis variant

The farewell message depends on the --mode flag used at runtime. In romantic mode it's warm, in shayari mode it's poetic, in toxic mode it's dramatic.

typing... typing

Flag What it does

VS Code Extension

Love Lang Tools is a full-featured VS Code extension for writing .love files. It provides syntax highlighting, IntelliSense, hover docs, a run button, and a visual settings panel.

Language ID: lovelang File ext: .love v1.0.0
Installing the Extension
Via CLI (recommended)
# Standard install
$ code --install-extension love-lang-tools-1.0.0.vsix --force
 
# Install to a specific profile
$ code --install-extension love-lang-tools-1.0.0.vsix --force --profile comp
Via VS Code UI
  • Open the Extensions panel (Ctrl+Shift+X)
  • Click the ··· menu at the top right
  • Select Install from VSIX...
  • Navigate to and select love-lang-tools-1.0.0.vsix
Building the Extension yourself
$ cd extension
$ npm run check
$ npm run package
# installs the generated .vsix
$ code --install-extension love-lang-tools-*.vsix --force
Extension Features
  • Syntax Highlighting — TextMate grammar covers declarations, operators, conditionals, loops, functions, festival blocks, modules, human-mode phrases, and modern built-ins.
  • Fallback Decoration Highlighting — If your theme doesn't color custom scopes, the extension applies its own decorations. Force-enable with lovelang.forceDecorationHighlighting = true.
  • IntelliSense & Snippets — Autocomplete for module syntax, default/named-arg patterns, collections, string toolkit, conversion helpers, filesystem functions, and human phrases.
  • Hover Documentation — Hover over any keyword to see its meaning, aliases, and usage.
  • Run Command — "Run Current File" button in editor title bar for .love files. Also available via Command Palette.
  • Settings Webview — Visual panel to configure all extension settings without editing JSON.
  • Auto-association — .love files automatically switch to Lovelang mode on open.

FAQ
Is Lovelang compiled or interpreted?
Interpreted. The pipeline is: file read → preprocess (human-phrase normalization) → lex → parse → tree-walk execute. There is no compilation step.
Can I mix natural-style and core syntax?
Yes, that's the intended workflow. You can write agar x bada hai 0 toh in one block and agar (x > 0) ye_karo { in the next. The preprocessor handles normalization before the lexer runs.
Does love you baby byeee actually stop execution?
Yes. The preprocessor normalizes it to love_byeee() which sets the runtime stop flag. Any statements after it in the same file do not execute.
What are removed features (streamlined mode)?
The following old keywords were removed in streamlined mode: dil, mood, junoon (old mutable), ishq (old const), and old output family babu_miss_you, shayarana, status, last_seen, seen, auto_reply, gaana. Built-ins chai_break and date_plan were also removed. If these appear, the parser reports an unknown statement.
What platforms does Lovelang run on?
macOS, Linux, and Windows. Helper run scripts are provided for all three platforms: run_love.sh (macOS/Linux), run_love.cmd (Windows CMD), run_love.ps1 (PowerShell).
Is there a loop guard?
Yes. The runtime automatically breaks out of a loop after 1,000,000 iterations with a warning. This prevents accidental infinite loops from freezing your terminal.
